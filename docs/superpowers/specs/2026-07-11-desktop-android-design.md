# Desktop and Android Design

## Scope

The product targets Windows, macOS, Linux and Android. iOS is explicitly outside the current scope. Desktop remains the reliable always-on node; Android is a foreground-real-time companion and full protocol participant while active.

Android does not request permanent foreground-service, accessibility, device-owner or default-keyboard privileges. Entering the background may suspend discovery, connections and clipboard observation. Returning to the foreground reconnects, requests a fresh `UiSnapshot`, then resumes incremental events.

## Shared architecture

- React UI, semantic design tokens and clipboard slot components are shared.
- Rust Core will own identity, groups, protocol validation, cache and Import state on every platform.
- Platform adapters isolate clipboard access, lifecycle, notifications, sharing and file-system differences.
- The existing `DesktopClient` boundary becomes a platform-neutral `AppClient`; UI commands remain revision checked.
- Network events only update remote device slots. Android clipboard writes still require an explicit user action and a valid Import transaction.

## Android lifecycle

```text
ColdStart / Resume
    -> Connecting
    -> SnapshotLoading
    -> ForegroundLive

ForegroundLive
    -> Pausing (application backgrounded)
    -> Suspended

Suspended
    -> Connecting (application foregrounded)
```

`Suspended` is not shown as an error. The UI retains a last-synchronized timestamp, marks remote slots as potentially stale and disables actions that require a live authorization check. Existing local cached content may only be imported if its authorization/Lease remains valid according to Rust Core.

## Android clipboard behavior

- While foregrounded, the application can observe supported clipboard changes after the platform permits access.
- A local clipboard change becomes the Android device's own slot automatically according to group policy.
- Imported remote content is suppressed and is not automatically republished.
- When Android denies or restricts clipboard access, the UI exposes capability state and a user-triggered “publish current clipboard” action instead of pretending capture is active.
- Text is implemented first. HTML, images, URIs and file lists are capability-gated additions.

## Android UI

- Bottom navigation replaces the desktop sidebar on narrow/touch layouts.
- Touch targets are at least 44 CSS pixels and do not rely on hover.
- The Android home page prioritizes current clipboard, active foreground state and device slots.
- The Switcher uses a full-screen sheet rather than a floating desktop overlay.
- Desktop-only concepts such as tray, global shortcuts and window-close behavior are hidden on Android.
- Android shows “foreground live”, “paused in background”, “reconnecting” and “clipboard permission limited” as distinct states.

## Sharing

The first mobile milestone includes an internal share-intent boundary and UI for received share payloads. Android manifest/Gradle integration for system Share Target is generated by Tauri Android initialization when an Android SDK is available. Shared text and URLs can be staged as a local capture; files and images require URI permission handling and bounded streaming before publication.

## Error handling

- Resume always requests a Snapshot; it never assumes the pre-background event revision is continuous.
- If local-network permissions, multicast, clipboard access or storage capability are unavailable, only the affected feature is disabled.
- Android battery optimization or process death is treated as normal suspension, not transfer failure.
- UI never claims background synchronization is active.

## Verification

- Shared TypeScript model and lifecycle reducer tests run without Android tooling.
- React tests verify Android navigation, foreground-state messages and explicit Import behavior.
- Production web assets build for both desktop and Android targets.
- Native Android initialization/build is conditional on Java, Android SDK/NDK and Rust Android targets being installed.
